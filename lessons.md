# inryokü — Lessons Learned

> Claudeが間違えるたびにここに記録する。同じ失敗を二度しないために。
> 追記方法: 「このルールをlessons.mdに書いて」と指示する。

## 2026-03 P1リビルド時の教訓

### L001: HOLD-TO-LOAD は廃止
**何が起きた**: mousedown長押しで進捗を進めるUIを実装したが、ブラウザの右クリックメニュー干渉、タッチデバイス非対応、進捗が戻るバグなど問題が多発。
**なぜダメ**: UXの複雑さとバグの温床。ユーザーが能動的に操作する必要があり体験を阻害する。
**代わりに**: AUTO_RATE[phase] による自動進行のみ。ユーザーは観るだけ。

### L002: ShaderMaterial.fragmentShader のランタイム書き換え禁止
**何が起きた**: フェーズに応じてfragmentShaderの文字列を書き換え、.needsUpdate = true で反映しようとしたが、WebGLコンテキストが不安定になり画面が黒くなった。
**なぜダメ**: Three.jsのShaderMaterialはコンパイル済みシェーダーを前提としており、ランタイム書き換えは想定外の動作。
**代わりに**: uniform変数で制御する。フェーズに応じてuniform値を変更し、シェーダー内のif/mix/stepで分岐。

### L003: OrthographicCamera + scissor 分割レンダーは複雑すぎる
**何が起きた**: 画面を左右分割（CMY白背景/RGB黒背景）するためにscissorレンダーを使ったが、座標計算が複雑でバグの特定が困難になった。
**なぜダメ**: デバッグコストが実装メリットを大幅に超える。非エンジニアの司さんが状況把握できなくなる。
**代わりに**: シンプルなPerspectiveCameraで統一。背景の分割はシェーダーやオーバーレイで表現。

### L004: フェーズ変数はrequestAnimationFrame内でのみ更新
**何が起きた**: setInterval内でphase変数を参照・更新したところ、rAFのフレームと同期せず古い値が残るバグが発生。
**なぜダメ**: setIntervalとrAFは独立したタイミングで実行される。phase変数の更新がフレーム描画と噛み合わない。
**代わりに**: 全てのphase更新はtick()関数（rAF）内で行う。外部タイマーからphaseを操作しない。

### L005: 大規模リファクタリングは1コミット1フェーズ
**何が起きた**: 複数フェーズの修正を一度にコミットした結果、P1全体が壊れた時にどの変更が原因か特定できなかった。
**なぜダメ**: ロールバックの粒度が粗すぎて、安定版に戻すのに大量の時間を消費。
**代わりに**: 1フェーズ実装→動作確認→コミット のサイクルを厳守。

### L006: P1コードが壊れたらまずgit logで安定コミットを探す
**何が起きた**: P1が壊れた時にパニックで修正を重ねた結果、さらに壊れて元に戻せなくなった。
**なぜダメ**: 壊れた状態の上に修正を重ねると、問題が複合化して原因特定が不可能になる。
**代わりに**: まず `git log --oneline` → 最後の安定コミットを特定 → `git diff` で何が変わったか確認 → 最小限の修正。

### L007: DOM要素のCSS管理とJS管理を混在させない
**何が起きた**: CSS transitionで opacity:0 にしたDOM要素を、後でJSの `style.opacity !== '0'` で判定しようとしたが、値が文字列 `'0'` に一致しなかった。
**なぜダメ**: CSS transitionとJS直接操作の間に状態の齟齬が生まれる。transition完了タイミングも不確定。
**代わりに**: DOMの表示制御はCSS transitionを使わず、全てJSのrequestAnimationFrame内でtransformとopacityをフレーム毎に計算して直接設定する。

### L008: 「実装しました」はブラウザで確認するまで信じない
**何が起きた**: コードを書いて「実装完了」と報告したが、実際のブラウザ表示に反映されていなかった。変数を更新するロジックは書いたが、render loopやDOM操作に正しく繋がっていなかった（例: overflow:hiddenでclip、display:noneで条件失敗、falloff計算でscale変数が輝度を潰す）。
**なぜダメ**: コードが存在することと、画面に反映されることは別。ShaderのUniform値を更新しても、別の計算がそれを打ち消す可能性がある。DOMのstyle値もCSS/JSの順序次第で期待通りにならない。
**代わりに**: 各修正後にconsole.logで「この変数が毎フレーム更新されているか」「この条件が通っているか」を確認してからコミット。render loop内で値が実際に変化していることを確認するまで完了と報告しない。
